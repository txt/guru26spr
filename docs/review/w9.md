# Glossary & Review Questions — March 25, 2026

## Glossary

### Lisp, Scheme, and Interpreters

**Lisp**
: List processing language (McCarthy, 1958). Code and data
  share the same representation: nested lists.

**Scheme**
: Purified subset of Lisp (Guy Steele, 1976 MIT thesis
  "Rabbit"). 5 keywords, 8 syntactic forms. Evaluates
  *all* arguments including the zeroth (function position).

**Lambda calculus**
: Alonzo Church's 1920s minimal model of computation.
  Theoretical foundation for Lisp, Scheme, and JavaScript.

**Prefix notation**
: Operator before operands: `(+ 1 2)` not `1 + 2`.
  Makes parsing trivial — just match brackets.

**S-expression**
: Either an atom (symbol or number) or a list of
  s-expressions. The universal Lisp data structure.

**Abstract syntax tree (AST)**
: Internal tree form of a program after parsing.
  In Lisp, already a nested list — parsing is almost free.

**Tokenizer**
: Breaks a character string into a list of tokens.
  Norvig's: add spaces around parens, then `str.split()`.

**Environment**
: A mapping `{symbol: value}` with a pointer to an outer
  environment. Variable lookup walks inner → outer.

**Lexical scoping**
: Variables resolve in the environment where the function
  was *defined*, not where it is *called*.

**Procedure (Norvig)**
: Three things: parameter names, a body expression,
  and the environment at definition time.

**`eval`**
: Recursively descend the AST: symbols → lookup, numbers →
  return, `if` → branch, `define` → store, else → call.

**REPL**
: Read-Eval-Print Loop. Lisp gave us interactive
  programming in 1963; COBOL/Fortran couldn't.

**First-class functions**
: Functions as values — pass them, return them, store
  them in variables. `(repeat twice)` returns a function.

**`quote`**
: Returns its argument literally, unevaluated.
  `(quote (+ 1 2))` ⇒ `(+ 1 2)`, not `3`.

**`set!`**
: Mutates an existing binding (vs. `define` which creates
  a new one in the innermost environment).

### Historical Context

**John McCarthy**
: Created Lisp (1958). Solved recursion via nested
  environments when hardware was stack-only.

**Peter Norvig**
: NASA → Google chief scientist. Co-wrote *the* AI
  textbook. Advocated Python over Lisp at Google.
  `lis.py` is 117 lines.

**Von Neumann memo (1948)**
: Defined computers as stack-based. Couldn't handle
  recursion — it took decades to work around.

**`car` / `cdr`**
: Contents of Address Register / Contents of Decrement
  Register. Named after the hardware, stuck forever.

**Smalltalk**
: Pure OO language. Everything is an object, including
  classes. Metaclass is an instance of class; class is
  an instance of metaclass — interpreter breaks the loop.

**JavaScript's origin**
: Built on LISP semantics (different syntax) in ~2
  weeks for Netscape. Lambda calculus made that possible.

**Lisp's legacy**
: Garbage collection, OS prototyping, multitasking, logic
  programming, interactive computing — all prototyped here.

### Software Process

**V-diagram**
: Plan → Code → Test. Planning informs test design;
  test results inform future planning.

**Vibe coding**
: Skip planning, ship fast, test on the client's time.
  Code review becomes retroactive planning.

### Package Management

**Package manager**
: Automates discovery, installation, dependency resolution,
  versioning, and removal of software.

**Dependency hell**
: A needs B needs C needs a different A. Recursive
  tarball unpacking was the 1990s experience.

**`pyproject.toml`**
: Single config file for modern Python: packaging,
  testing, linting, formatting. Replaces `setup.py`,
  `setup.cfg`, `requirements.txt`, and 3+ `.ini` files.

**Wheel (`.whl`)**
: Distribution format — a ZIP of code + metadata.
  `py3-none-any` = Python 3, no C, any platform.

**PyPI**
: Python Package Index. ~750K projects, 8M releases.
  Names are globally unique, first-come-first-served.

**`pip install -e .`**
: Editable install — symlinks source so changes take
  effect immediately. The dot means "this directory."

**`[project.scripts]`**
: Maps CLI command names to Python functions.
  `ezr = "ezeg:cli"` → typing `ezr` calls `cli()`.

**Virtual environment**
: Isolated `site-packages` per project. Python's answer
  to "Project A needs numpy 1.24, Project B needs 2.0."

**`twine`**
: Uploads built packages to PyPI. Prompts for an API
  key (expires every 30 days).

**`__all__`**
: Controls what `from X import *` exports. Names with
  leading `_` excluded by default unless listed.

**Left-pad incident (2016)**
: 11-line npm package unpublished → global JS builds
  broke. Backstory: author was legally threatened over
  the package name.

**Reproduction package**
: Code + data + scripts bundled with a paper. Only ~25%
  actually work in practice (outdated deps, too slow).

**`test.pypi.org`**
: Staging server for PyPI. Push here first so mistakes
  don't go global.

**Optional dependencies**
: `pip install ezr[test]` installs pytest;
  `pip install ezr` doesn't. Keep core installs lean.

**Pre-commit vs. post-commit hooks**
: Auto-generated files (docs) should be pre-commit to
  temp, not post-commit to repo — avoids collisions.

## Review Questions

**1.**
(a) What are the five syntactic forms of Norvig's Lispy
    Calculator, and which two are "special forms"?
(b) Scheme has 5 keywords and 8 syntactic forms; Python
    has 33 and 110. What practical consequence does this
    difference have for writing tools that reason about
    programs (e.g., refactoring tools, formal verifiers)?

**2.**
(a) What does Norvig's `eval` do when it encounters a
    bare symbol like `r`?
(b) Explain why the `Env.find` method must walk outward
    through nested environments. What would go wrong if
    `eval` only searched the innermost environment?

**3.**
(a) In Scheme, what is the difference between `define`
    and `set!`?
(b) The `make-account` example creates closures with
    mutable state via `set!`. How does lexical scoping
    make this "bank account" pattern work, and why would
    it fail under dynamic scoping?

**4.**
(a) Why was recursion difficult to implement on hardware
    built according to the von Neumann memo?
(b) McCarthy solved recursion via nested environments in
    1958, yet Fortran didn't support it until 1995. What
    does this ~37-year gap tell us about how hardware
    assumptions constrain language design?

**5.**
(a) Name two technologies first prototyped in Lisp that
    are now universal in modern computing.
(b) The lecture claimed JavaScript was built on Scheme's
    semantics in about two weeks. What specific property
    of lambda calculus made such rapid implementation
    feasible, and what tradeoffs did that speed create?

**6.**
(a) What are the "five headaches" every package manager
    must address?
(b) The left-pad incident broke builds worldwide over 11
    lines of code. Propose a package management feature
    that would prevent this, and argue what new problems
    your solution introduces.

**7.**
(a) What does `pip install -e .` do differently from
    `pip install ezr`?
(b) A team of five is developing a Python package. Two
    members change `ezr.py` on the same day. Explain
    how the distinction between editable installs, the
    Git repo, and the PyPI package prevents (or fails
    to prevent) conflicts at each stage.

**8.**
(a) Why does `pyproject.toml` list `pytest` under
    optional dependencies rather than regular
    dependencies?
(b) A user reports that `pip install ezr` also installs
    `ezeg.py` into their `site-packages`. Argue whether
    test/tutorial code should or should not ship with the
    package, using the "repo = development, package =
    usage" distinction from the lecture.

**9.**
(a) What does the `__all__` list control, and why did
    `ezr.py` need to add `_nest` and `_cluster` to it?
(b) Auto-generated documentation committed by a
    post-commit hook caused merge collisions in Tim's
    workflow. Explain his proposed fix (pre-commit to
    temp, one librarian copies to repo) and identify a
    scenario where even this fix breaks down.

**10.**
(a) Tim recommends working with LLMs generated code in chunks of ~100
    lines or less. What failure mode did he observe
    with larger chunks?
(b) The lecture described LLM failures on a Bayes
    classifier m-estimator, a Taylor expansion, and a
    CDF symmetry problem — all textbook material. Using
    the V-diagram concept, argue where in the
    plan-code-test cycle AI-generated code most needs
    human oversight, and why "vibe coding" makes that
    oversight harder.
