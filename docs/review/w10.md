### **Glossary: Review Vocabulary & TLAs**
These terms form a shared language used to compress complex design feedback during 
the review process.

* **DRY (Don't Repeat Yourself):** Avoid logic duplication; having the same code in 
  two places means two places to fix.
* **KISS (Keep It Simple, Stupid):** Prioritize simple, readable solutions over 
  "clever" or overly complex ones.
* **YAGNI (You Ain't Gonna Need It):** Do not build features or abstractions for 
  imagined future requirements.
* **SRP (Single Responsibility Principle):** A module, class, or function should 
  have exactly one reason to change.
* **OCP (Open/Closed Principle):** Software entities should be open for extension 
  but closed for modification.
* **LSP (Liskov Substitution Principle):** Subclasses must be substitutable for 
  their base classes without breaking the system.
* **ISP (Interface Segregation Principle):** Small, focused interfaces are better 
  than "fat," general-purpose ones.
* **DIP (Dependency Inversion Principle):** Depend on abstractions (interfaces) 
  rather than concrete implementations.
* **LoD (Law of Demeter):** Only talk to immediate neighbors; each "dot" in 
  `a.b.c.d()` is a hidden dependency.
* **CQS (Command Query Separation):** A method should either change state or return 
  a value, but not both.
* **POLA (Principle of Least Astonishment):** Code should behave in a way that is 
  least likely to surprise the reader.
* **TDA (Tell Don't Ask):** Tell objects what to do; do not interrogate their 
  internal state to make decisions.
* **SLAP (Single Level of Abstraction):** Keep all code within a single function 
  at the same level of detail.
* **AHA (Avoid Hasty Abstractions):** Do not DRY too early; duplication is often 
  cheaper than the wrong abstraction.

### **Quiz Questions**

**Q1: The Scope of Review**
* **a)** What specific non-source-code artifacts are now commonly included in 
  modern Pull Requests?
* **b)** Why does the "Everything Review" framework argue that a YAML configuration 
  error is as critical as a Python logic bug?

**Q2: The Purpose of Review**
* **a)** According to the Rigby & Bird (2013) study, what is the primary measurable 
  benefit of code review?
* **b)** Explain how code review serves as a mechanism for building "shared 
  ownership" within a software team.

**Q3: AI and the Review Bottleneck**
* **a)** How has the use of LLMs changed the ratio of code written to code reviewed 
  for the average developer?
* **b)** Identify two types of errors or contexts that human reviewers catch which 
  AI currently struggles to identify.

**Q4: PR Size and Complexity**
* **a)** What is the recommended maximum size for a PR, and what happens to review 
  quality when this limit is exceeded?
* **b)** Why is review complexity described as "quadratic" regarding the number 
  of new functions added to a codebase?

**Q5: Reviewer Ethics and Efficiency**
* **a)** What is the functional difference between a "Draft" PR and a "Published" 
  PR in terms of reviewer notification?
* **b)** How does adhering to a short, readable style guide serve as infrastructure 
  for a more efficient review process?

**Q6: Historical Context (Fagan Inspections)**
* **a)** What were some of the defined roles in a heavyweight Fagan inspection 
  meeting?
* **b)** Why did the industry move away from Fagan's heavyweight model toward the 
  "contemporary" practices studied by Rigby?

**Q7: Perspective-Based Reading (PBR)**
* **a)** Name three distinct stakeholder perspectives used in PBR to guide a 
  reviewer's focus.
* **b)** How does the "procedural guidance" of PBR differ from a standard 
  "free-reading" review style?

**Q8: Industrial Findings (Google and Microsoft)**
* **a)** Approximately how many hours per day do developers at Google spend on 
  code review?
* **b)** Contrast the "stated motivation" for review (finding bugs) with the 
  "actual practice" found in the Microsoft study.

**Q9: Design Principles in Review**
* **a)** Define the "Principle of Least Astonishment" (POLA) as it relates to 
  code readability.
* **b)** Provide an example of a "Tell Don't Ask" (TDA) violation and explain why 
  it is considered poor design.

**Q10: Advanced Review Tooling**
* **a)** What is a "Stacked PR," and what tool was mentioned to help manage them 
  on GitHub?
* **b)** Evaluate the role of LLMs as a "pre-filter" in the modern review 
  pipeline.
