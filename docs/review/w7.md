# Glossary + Review Questions: Mar 4 Lecture

---

## Glossary

### Industry Context & AI Economics

**Power prompt**
  A refined, multi-hundred-line prompt that is itself the
  deliverable; shared between developers.

**Prompt as product**
  The idea that crafting and iterating prompts (e.g. 450
  lines for loop invariants) is real engineering output.

**Loop invariant**
  A property that remains true before and after every
  iteration of a loop; central to formal verification.

**Theorem prover**
  A tool that mechanically checks or generates proofs of
  correctness (e.g. for Rust code).

**Vibe coding**
  Hands-free prompting where you keep feeding an LLM until
  the output "looks good"; industry view: "vibe is chaos."

**Nonfunctional requirements**
  Qualities beyond correctness: security, accessibility,
  maintainability, performance.

**Accessibility (web)**
  Design so disabled users can navigate; includes alt tags,
  colorblind-safe palettes, screen-reader support.

**Alt tags**
  Text descriptions on web elements so screen readers can
  describe them to visually impaired users.

**Male colorblindness**
  ~8% of males cannot distinguish red/green or green/yellow;
  a vibe-coded site may ignore this.

**$800B gap**
  Projected 2030 shortfall between AI infrastructure costs
  and expected revenue; bigger than any government bailout.

**Post-boom AI**
  The anticipated correction when venture capital retreats
  from heavily subsidized AI services.

**Product manager**
  Translates technical to client and back; growth area
  (~4x/year); asks "what should we build?"

**Henry Ford / faster horse**
  Customers describe incremental wants; experts see
  transformative solutions (vaccine > iron lung).

### Review: RAG, Hallucinations, DSLs, FSMs

**RAG (Retrieval-Augmented Generation)**
  Index background docs, retrieve relevant chunks, inject
  them alongside the query into the LLM.

**Chunking**
  Splitting a corpus into small pieces (~100 words) for
  embedding and retrieval.

**Hallucination**
  An LLM fabricates plausible but nonexistent content
  (e.g. fake paper titles for real authors).

**Hallucination taming**
  Test everything; query multiple LLMs; repeat queries and
  average; ground with RAG.

**Paging analogy**
  RAG context window = RAM (fast, small); corpus = disk
  (slow, large); chunks = pages swapped in.

**Engine / rules / payload**
  Engine: small interpreter (5 lines). Rules: domain model
  the user writes. Payload: runtime data carried around.

**Elbow test**
  A DSL succeeds when users can elbow the developer out of
  the way and write their own rules.

**Finite state machine (FSM)**
  States + transitions + events. Distinguished start state,
  distinguished exit state.

**Livelock**
  Enter a cycle with no exit; the lollipop — stick goes in,
  loop has no way out.

**Deadlock**
  Reach a state with no outgoing transitions; stuck with
  no way forward.

**Transitive closure**
  Matrix operation (O(n³)) to find everything reachable
  downstream from a node in a directed graph.

### Smalltalk

**Alan Kay**
  "The best way to predict the future is to invent it."
  Creator of Smalltalk at Xerox PARC.

**Xerox PARC / Alto**
  Birthplace of bitmap graphics, the mouse, windows, and
  Smalltalk; portrait screen for word processing.

**Steve Jobs and PARC**
  Jobs saw the Alto demos, bought decades of research for
  ~$1M, pivoted Apple toward GUIs.

**Inversion of control**
  OO principle: instead of passing data to controllers,
  pass controllers (blocks) to data.

**Message send**
  Smalltalk's only operation: a receiver gets a message
  and decides how to respond.

**Unary message**
  No arguments: `x size`, `x printString`.

**Binary message**
  One argument, symbolic: `3 + 4`, `10 > 7`.

**Keyword message**
  Named arguments: `array at: 2 put: 99`. Reads like
  a sentence.

**Precedence**
  Unary > binary > keyword. Within a kind: left to right.
  No BODMAS — use parentheses.

**Block (first-class function)**
  `[ :x | x + 1 ]` — a sleeping beauty. Woken by sending
  `value` or `value:`.

**True / False as classes**
  Boolean subclasses True and False; each a singleton.
  `ifTrue:` on True evaluates the block; on False, does
  nothing (empty method body).

**Fluent programming**
  Methods return `self` by default, so messages chain:
  `Counter new initialize inc inc value`.

**Smalltalk class hierarchy**
  Object → Boolean → True/False;
  Object → Magnitude → Number → Integer → SmallInteger.

**Stream / Random**
  Random is a subclass of Stream; `atEnd` always returns
  false — infinite sequence of numbers.

**Image-based persistence**
  Smalltalk writes entire RAM to an image file on exit;
  resume exactly where you left off. Makes sharing harder.

**`collect:` / `select:` / `do:`**
  Collection messages: map, filter, iterate. The collection
  owns the loop, not the caller.

**`:=` (assignment)**
  Colon-equals assigns; plain `=` compares. Period ends a
  statement (not semicolon).

**GNU Smalltalk (GST)**
  Command-line Smalltalk; avoids the windowed IDE; files
  loaded via `FileStream fileIn:`.

**Ruby's debt to Smalltalk**
  Blocks, message-based iteration, garbage collection.
  Java also borrowed GC and single inheritance from ST.

### Prolog

**Logic programming**
  Declare what is true and what follows; let the engine
  handle the search.

**Atom**
  Lowercase constant: `tim`, `banana`, `parent`.

**Variable**
  Uppercase box to be filled by unification: `X`, `Y`.

**Unification (matching)**
  `X = tim` puts the symbol `tim` into the box `X`;
  not assignment — pattern matching.

**Fact**
  A ground assertion ending with `.`:
  `parent(tim, pat).`

**Rule**
  A clause with `:-` (read "if"):
  `grandparent(X,Z) :- parent(X,Y), parent(Y,Z).`

**Head / Tail of a list**
  `[H|T]` splits a list: `[a,b,c]` → `H=a`, `T=[b,c]`.
  Recursive structure of nested pairs.

**`member/2`**
  X is in the list if it is the head, or if it is in the
  tail. Two clauses; generates on backtracking.

**`append/3`**
  Append [] to L gives L; append [H|T] to L gives
  [H|append(T,L)]. Runs in any direction.

**Bidirectionality**
  One relation, many uses: `append(X,Y,[a,b,c])` returns
  all ways to split [a,b,c]. No separate inverse function.

**Bratko cardiac diagnosis**
  Input EKGs → diagnosis; input diagnosis → expected
  measurements; partial instantiation → "what if" queries.

**Backtracking**
  Prolog keeps a stack of choice points and bindings;
  semicolon (`;`) retreats to the last choice point.

**Base case / recursive case**
  Every recursive predicate needs a termination clause
  and a clause that calls itself on something smaller.

**Monkey and bananas**
  Classic state-space search: walk, push, climb, grasp.
  State = `state(MonkeyPos, BoxPos, Height, HasBanana)`.

**Depth-first search (DFS)**
  Explores one branch fully before trying alternatives.
  Low memory; may miss short solutions.

**Breadth-first search (BFS)**
  Explores level by level; finds shortest solutions first
  but memory of the frontier explodes.

**Depth-first iterative deepening (DFID)**
  DFS with a leash: try depth 1, then 2, then 3, ...
  Finds short solutions with DFS memory cost.

**Dog-on-a-leash analogy**
  Leash length 10 vs 1: area ratio is r², so repeating
  shorter searches adds only ~1% to total cost.

**Korf 1985**
  35-page AAAI paper proving DFID is admissible; the
  math reduces to 2 lines of Prolog.

**Branching factor threshold**
  DFID overhead drops below 1% when branching factor
  exceeds ~2.25 (Korf's result).

**Loop avoidance**
  Track visited states; `\+ member(S1, Seen)` prevents
  revisiting in the search.

**SQL as cut-down logic programming**
  SQL queries return sets like Prolog; but SQL cannot do
  recursion in rules.

**Fifth Generation Project**
  Japan's 1980s plan to build Prolog hardware for AI;
  triggered American backlash favoring Lisp.

**Answer set programming (ASP)**
  Ground a Prolog program, then solve; negation becomes
  easy. "What the cool kids use."

**Constraint logic programming (CLP)**
  Variables carry constraints (e.g. `X #\= Y`); the
  solver prunes impossible values during search.

**Z3 / PicoSAT**
  Modern theorem provers and SAT solvers; predicted to
  become part of everyday SE workflows.

### DSL Methodology (Expanded)

**Tape recorder method**
  Record experts and novices; find idioms experts use
  that novices do not; those become DSL keywords.

**Language compression**
  Train on 10 paragraphs to design the language; if the
  remaining 100 code up easily, you have the right DSL.

**Expert chunking (chess fork)**
  Experts see high-level patterns (fork, pin) where
  novices see individual pieces on 64 squares.

**Consulting timeline**
  Weeks 1-3: record domain language. Week 4: show users
  the language. Week 8: users write most of it.

### Q7 — Reading Prolog: Family Relations

Given this database and rule:

```prolog
parent(tim, pat).
parent(pat, ann).
parent(pat, bob).
parent(ann, liz).
parent(bob, sam).

ancestor(X, Y) :- parent(X, Y).
ancestor(X, Z) :- parent(X, Y), ancestor(Y, Z).
```

**(a)** Which clause is the base case and which is the
recursive case? What does `:-` mean?  *(2 min)*

**(b)** Trace the query `ancestor(tim, X).` step by
step. Show which clause fires first, what `X` binds to,
and what happens on each backtrack (`;`). List every
value of `X` that Prolog will return. Then explain: if
we swapped the two `ancestor` clauses so the recursive
one came first, what could go wrong?  *(4 min)*

### Q8 — Reading Prolog: Monkey and Bananas

Given this code:

```prolog
move(state(P, P, floor, H), climb,
     state(P, P, onbox, H)).

move(state(P1, P1, floor, H), push(P1,P2),
     state(P2, P2, floor, H)) :-
  place(P1), place(P2), P1 \= P2.

move(state(middle,middle,onbox,no), grasp,
     state(middle,middle,onbox,yes)).
```

**(a)** In the `climb` clause, why does `P` appear four
times? What stays the same and what changes between
the input and output states?  *(2 min)*

**(b)** In the `push` clause, explain what `P1 \= P2`
enforces. Identify every field that changes and every
field that stays invariant. Then look at `grasp`:
list all the preconditions the monkey must satisfy
before grasping, and explain why `\+ member(S1, Seen)`
is needed in the search engine that calls these
moves (connect to livelocks in FSMs).  *(4 min)*
