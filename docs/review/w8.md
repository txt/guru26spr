# W8 review - Vibe Coding Review


## Part 1: Glossary of Concepts

* **Vibe Coding**  
  Vibe coding is the practice of building software by describing what you want in natural language and using an LLM or code-generation model to produce the implementation. The key idea is that the programmer is no longer spending most of their time typing code line by line. Instead, the programmer is guiding the system, evaluating its output, refining the result, and deciding what should be kept or rejected. This is not just about speed. It changes the role of the engineer from primarily writing code to directing and reviewing generated code.

* **Abstraction Shift**  
  Software engineering has always moved upward through layers of abstraction. Programmers once worked much closer to hardware, then moved to higher-level languages, then to frameworks and reusable platforms. Vibe coding is another step in that same pattern because it allows developers to express intent at an even higher level. But each new abstraction makes understanding the underlying system more important, not less. You may write less code directly, but you still need to understand what the system is doing and whether the generated solution is actually sound.

* **Judgment**  
  In a vibe-coding workflow, the main bottleneck is no longer producing code quickly. The real challenge is deciding whether generated code is correct, simple, maintainable, secure, and appropriate for the actual problem. This is what the lecture means by judgment. A person with strong judgment can recognize when code only appears to work, when it introduces unnecessary complexity, or when it solves the wrong problem in a polished way. Since LLMs can generate large amounts of code very quickly, the value of human judgment increases because bad decisions can now be scaled just as easily as good ones.

* **Prompt Precision**  
  Prompting is not just casual conversation with a chatbot. In this context, it works like a lightweight software specification. The quality of the result depends heavily on how clearly the engineer states the goal, the constraints, the architecture, the technologies, and what should be excluded. A vague prompt often leads to vague or overly elaborate output. A precise prompt gives the model boundaries and helps produce code that is more aligned with the actual need. This means prompt writing is an engineering skill because it requires clear thinking about requirements, tradeoffs, and scope.

* **Core Engineering Principles under LLMs**  
  The lecture emphasizes that principles like **DRY**, **YAGNI**, and **Separation of Concerns** do not disappear in vibe coding. In fact, they become even more important because models often generate code that looks polished while quietly violating them. An LLM may duplicate logic across files, add unnecessary abstraction or extra features that were never requested, or mix UI, business logic, data handling, and configuration into one place just to make something “work.” These principles remain important because they help engineers detect and remove the kinds of structural problems that AI systems are especially good at producing quickly.

* **VITAL and Simplicity**  
  VITAL, in the sense used in the lecture, points toward building systems that are simpler, more controllable, and less dependent on unnecessary external complexity. This is why the lecture values constraints such as a single HTML file, no framework, no build process, and easy static deployment. These are not random limitations; they are ways of forcing discipline. Vibe coding can generate complicated systems very easily, but ease of generation does not make that complexity worthwhile. The lecture’s point is that simple solutions are often better because they are easier to inspect, understand, maintain, and own.

* **Skeptical Acceptance**  
  A healthy mindset toward AI-generated code is neither blind trust nor total rejection. The lecture recommends skeptical acceptance: treating generated output as potentially useful, but never automatically reliable. AI systems can confidently produce incorrect logic, invent APIs, misunderstand constraints, or generate code that works only in a narrow case. Because of that, generated code should be treated like a draft or a pull request from someone who writes quickly but does not fully understand the broader system. It can be helpful, but it must still be reviewed carefully and understood before it is accepted.

2. **Why does the lecture argue that judgment matters more than coding speed in an LLM-based workflow?**

6. **What does “skeptical acceptance” mean, and why is it a better mindset than either blindly trusting or rejecting AI output?**

7. **Why can faster code generation actually increase risk in software projects?**

8. **Why is prompt writing an engineering skill, not just a chatbot trick?**  
   Explain how constraints improve outcomes.

9. **Why does the lecture connect vibe coding with VITAL and simplicity?**  
   Explain why simple, local, low-dependency solutions still matter.
