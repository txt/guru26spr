## Glossary

### RAG & Virtual Memory

**RAG (Retrieval-Augmented Generation)**
  Feed an LLM only the retrieved chunks relevant to a query,
  not the whole corpus.

**Context window**
  The LLM's "RAM" — small, fast, expensive; must be curated.

**Corpus**
  The full document collection — the "disk"; large but slow to
  search exhaustively.

**Semantic paging**
  Retrieving only the relevant text "pages" into the LLM context,
  mirroring OS virtual-memory paging.

**Chunk**
  A fixed-size excerpt (~150–220 words) used as the retrieval
  unit. Trades recall (big) vs. precision (small).

**Embedding**
  A vector in ℝᵈ encoding the meaning of a text chunk.
  Distributional semantics: similar meaning → nearby vector.

**page.file**
  Serialised cache: FAISS index + embeddings + chunks +
  manifest. Analogous to a swap file or Make cache.

**Manifest**
  Dict of {filepath: mtime+size hash}; enables cache
  invalidation without re-encoding unchanged PDFs.

**Cache invalidation**
  Manifest mismatch → evict and reload. Same pattern as Make,
  Webpack, ccache.

**Grounding vs. hallucination**
  Without context the LLM interpolates from priors (confabulates).
  Retrieved chunks constrain generation to "synthesise, not recall."

**Semantic Page Table**
  Debug output: for each retrieved chunk, prints rank, L2²,
  doc/page, and a snippet. Auditability.

**Incremental update**
  Encode only new/changed chunks; append to existing index.
  Full rebuild needed only on deletions (IndexFlatL2 lacks
  efficient deletes).

### DSLs & System Architecture

**DSL (Domain-Specific Language)**
  A very high-level notation tailored to one problem area;
  learnable by a domain expert in < 1 day. E.g. SQL, AWK, regex.

**External DSL**
  Parse a string; interpret it. Examples: SQL, dot files, regex.

**Internal DSL**
  Exploit host-language features (subclassing, tables) to look
  domain-like. Cheaper to build.

**Elbow test**
  Do domain users elbow the programmer aside to fix obvious
  errors in the code? If yes, the DSL is readable enough.

**Accidental complexity**
  Domain logic buried under infrastructure; the *what* entangled
  with the *how*.

**Engine / Rules / Payload**
  Universal three-layer DSL architecture:
  - Engine: how time passes and state updates (universal)
  - Rules: domain knowledge — stocks, flows, transitions (situated)
  - Payload: mutable instance data (particular)

**Stock**
  A quantity that accumulates over time (compartmental models).

**Flow**
  A rate that changes a stock per time unit.

**Compartmental model**
  Stock-and-flow simulation; used for ecology, economics,
  software project management (e.g. Brooks's Law).

**Separation of concerns**
  Engine is invisible to domain expert; rules contain zero
  infrastructure. The engine calls the rules; never vice-versa.

**Finite State Machine (FSM)**
  States + events + transitions + actions. Same engine/rules/
  payload pattern as compartmental models.

**Payload injection**
  FSM gains power by rewriting its own event queue
  (e.g., `inject(p, "die")` when hp ≤ 0).

**Idiom extraction**
  DSL design process: identify repeated clumsy patterns in raw
  code → name them → write a helper → the rules become readable.

**Observability**
  Adding trace/lint without modifying the engine (Q1–Q2 pattern).

**DOT export**
  Because the rules table *is* the model, you can read it as
  data and generate Graphviz diagrams, tests, or coverage reports.

## Review Questions

### Q1 — Virtual Memory Analogy

**a)** In one sentence each, define: context window, corpus,
chunk, and retriever — using the OS paging metaphor.  *(2 min)*

**b)** A colleague proposes sending the entire PDF corpus as
context to every LLM query. Explain two concrete failure modes
and describe how RAG resolves each.  *(4 min)*

---

### Q2 — Embeddings

**a)** What is a sentence embedding, and why do semantically
similar sentences have nearby vectors?  *(2 min)*

**b)** You query "automobile safety recalls". Your corpus uses
the word "car", never "automobile". Explain why a well-trained
embedding model retrieves relevant chunks anyway, and what
would break this guarantee.  *(4 min)*

---

### Q3 — Grounding vs. Hallucination

**a)** Define hallucination in an LLM and explain in one
sentence how injecting retrieved chunks reduces it.  *(2 min)*

**b)** A RAG system correctly retrieves relevant chunks but
the LLM still produces a wrong answer. Give two distinct
reasons this can happen and propose one mitigation for each.
*(4 min)*

---

### Q4 — Cache Invalidation

**a)** What fields does the RAG manifest store, and what does
a cache *miss* trigger?  *(2 min)*

**b)** A user deletes two PDFs and adds three new ones.
Walk through exactly what `update_pagefile()` does in each
case (deletion vs. addition) and explain *why* deletions
force a full rebuild with IndexFlatL2.  *(4 min)*

---

### Q5 — Chunking Trade-offs

**a)** What is the empirically robust chunk size in words,
and what two properties does it balance?  *(2 min)*

**b)** You increase chunk size from 180 to 800 words.
Describe the effect on retrieval precision, context-window
pressure, and hallucination risk. Then argue for or against
the change for a legal-document QA system.  *(4 min)*

---

### Q6 — The Elbow Test

**a)** State the elbow test in one sentence. What is the
consequence of failing it for long-term maintenance?  *(2 min)*

**b)** You show your FSM rules table to a game designer with
no programming background. They immediately spot a missing
transition and fix it themselves. Explain what DSL property
made this possible, and contrast it with a system where the
same logic is buried in a 200-line `run()` function.  *(4 min)*

---

### Q7 — Engine / Rules / Payload

**a)** Name the three layers of the generalised DSL
architecture and state who *owns* each layer.  *(2 min)*

**b)** Compare the compartmental model (`Model.run()` /
`Diapers.step()`) and the FSM (`fsm3.run()` / `rules` table)
on all three layers. What does this structural similarity
tell you about DSL design in general?  *(4 min)*

---

### Q8 — External vs. Internal DSL

**a)** Distinguish external from internal DSLs with one
example of each from lecture.  *(2 min)*

**b)** A team proposes writing a custom parser for an
external DSL instead of using Python subclassing. List
two advantages and two disadvantages of the external
approach, then recommend one for a project where the
domain expert is a statistician with no programming
background.  *(4 min)*

---

### Q9 — Rules as Data

**a)** What does it mean for the rules table to be "first-class
structure" rather than just code? Give one artifact you can
generate from it.  *(2 min)*

**b)** A team argues their FSM rules should be compiled into
the engine for performance. Explain what they lose by doing
this, using the concepts of auditability, separation of
concerns, and the elbow test. When, if ever, would the
trade-off be worth it?  *(4 min)*

---

### Q10 — RAG vs. Fine-tuning

**a)** In one sentence, state what RAG does that a plain
LLM cannot. What is the LLM's job reduced to once chunks
are injected?  *(2 min)*

**b)** A startup proposes fine-tuning a model on their
internal PDFs instead of building a RAG pipeline. Compare
the two approaches on: update latency (new PDFs arrive
daily), auditability (which source generated the answer),
and cost. Recommend one and justify.  *(4 min)*
